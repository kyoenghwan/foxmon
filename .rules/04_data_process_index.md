# Part 4: 전방위 UI 색인 맵 & 파급 범위 제어 프로토콜

## 4.1 전방위 AI용 프로젝트 색인 맵 (핵심 기능)

모든 UI 요소, 페이지, 원자(8-Atom)의 위치와 관계를 단일 파일(`docs/project_map.yaml`)에서 관리하는 **SSOT(Single Source of Truth)** 시스템입니다. AI는 어떤 작업을 수행하기 전에 반드시 이 파일을 먼저 조회해야 하습니다.

> **[AI 절대 지침: 수동 템플릿 참조]**
> `project_map.yaml` 요소 등록 또는 `style_group`, `project_rule` 추가 등 색인 구조를 조회하거나 생성할 때, **반드시 터미널/파일 도구(`view_file`)를 통해 `docs/templates/project_yaml_templates.md` 문서를 여십시오.** 임의로 구조를 상상하여 작성하는 것은 금지됩니다.

---

## 4.2 AI 동작 프로토콜 (핵심 4단계)

### 4.2.1 표준 UI 수정 프로세스
1. **키워드 검색**: `docs/project_map.yaml`에서 대상 요소 검색
2. **파급 범위 확인**: `style_group` 존재 시 "단일 vs 그룹 전체" 사용자 선택
3. **특수 지침 적용**: `project_rule` 존재 시 해당 지침 파일 우선 로드
4. **색인 누락 대응**: 대상이 없으면 색인 엔트리 생성 → 승인 → 코드 작업

### 4.2.2 필수 확인 메시지 형식
~~~text
📌 색인 확인 결과:
- 요소명: [ELEMENT_NAME]
- 위치: [파일경로] ([selector])  
- 스타일 그룹: [STYLE_GROUP/없음]
- 프로젝트 특수 지침: [PROJECT_RULE/없음]

이 대상을 수정하는 것이 맞습니까? (예/아니오)
~~~

## 4.3 색인 생성/업데이트 강제 규칙

### 4.3.1 필수 업데이트 상황
다음 상황에서는 **반드시 색인 맵을 먼저 업데이트**하고 사용자 승인을 받은 후 코드 구현:
- 새 페이지/모달/주요 UI 섹션 생성
- 새 주요 UI 요소(버튼, 텍스트, 메뉴, 입력 필드) 생성  
- 컴포넌트 파일 경로나 CSS selector 변경
- Flow Atom(FA), Trigger Atom(TA), Event Atom(EA) 추가/변경
- 공유 스타일 그룹(`style_group`) 또는 프로젝트 지침(`project_rule`) 구조 변경

### 4.3.2 3단계 절차
**Step 1**: 색인 엔트리 초안 작성 → **Step 2**: 사용자 승인 요청 → **Step 3**: 승인 후 코드 구현

## 4.4 DB 스키마 관리 프로토콜 (SSOT)

### 4.4.1 핵심 원칙
- **경로**: `docs/db/[테이블명]_schema.md`
- **SSOT 원칙**: DB 필드명은 반드시 이 문서 기준, 임의 추측 절대 금지
- **수정 프로세스**: 스키마 문서 업데이트(Version 증가) → 승인 → DA/QA/OA 코드 수정

### 4.4.2 템플릿 참조 의무
> **[AI 절대 지침: 스키마 양식 참조]**
> 스키마 문서 작성 시 반드시 `view_file`로 `docs/templates/project_yaml_templates.md`의 양식을 복사하여 사용하십시오.

---

## 4.5 프로세스 청사진 프로토콜

- **작성 시점**: 새 도메인/큰 기능 개발 시 코드보다 먼저 작성
- **경로**: `docs/processes/[프로세스명]_process.md`
- **템플릿 의무**: 반드시 `view_file`로 템플릿 파일의 양식을 복사하여 UI 흐름, 8-Atom 연결, 트랜잭션 구간 명시

---

## 4.6 프로젝트 특수 지침 시스템

### 4.6.1 우선순위 체계
- **절대 불변**: 8-Atom 체계 순수성
- **최우선**: `docs/project_rules/` 특수 지침
- **기본**: 범용 지침(00~04)

### 4.6.2 AI 자동 제안 트리거
다음 상황 감지 시 프로젝트 특수 지침 생성을 자동 제안:
- 여러 컴포넌트의 동일 제약 공유 필요
- "항상 같이 움직여야 해" 류의 도메인 제약 언급
- 범용 원칙을 넘는 엄격한 규칙의 반복 필요

### 4.6.3 템플릿 참조
프로젝트 개요 및 도메인 규칙 작성 시 `project_yaml_templates.md` 양식 준수 필수

## 4.7 반응형 및 다국어 대응 원칙

- **반응형**: `responsive_variants` 존재 시 뷰포트 선택 필수 (모바일/데스크톱/전체)
- **다국어**: `i18n_key` 존재 시 "한국어만 vs 모든 언어 공통" 선택 필수
- **프로젝트 지침 우선**: 특수 지침의 반응형 규칙이 있으면 해당 지침 최우선 적용
