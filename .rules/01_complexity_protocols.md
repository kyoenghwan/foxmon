# Part 1: 복잡도 분석 & 3단계 프로토콜

## 1.1 자동 복잡도 분석 알고리즘

기능 추가/수정 요청 시 AI는 코딩 전 내부적으로 복잡도 점수를 계산합니다.

```yaml
complexity_scoring:
  high_complexity_indicators: # 각 10점
    - "저장", "조회", "계산", "검증", "결제", "이메일", "권한", "승인"
  medium_complexity_indicators: # 각 5점
    - "필터", "조건", "분기", "페이징"
  structural_complexity:
    database_access: 15점
    external_api_call: 10점
    authentication_required: 10점
    authorization_required: 10점
    transaction_required: 15점
```

## 1.2 행동 결정 기준

- **정적 원자 제한 예외 (DA/CA/TA/EA)**: 항상 복잡도 0점으로 간주 (즉시 코드 생성)
- **0~9점 (단순 UI/텍스트)**: 색인 타겟 확인 후 즉시 구현합니다.
- **10점 이상 (비즈니스 로직)**: "비즈니스 로직이 포함되어 설계가 필요합니다" → **YAML 설계 강제**

## 1.3 3단계 강제 실행 프로토콜 (10점 이상 시)

복잡도 산정 결과 10점 이상인 경우, 아래 3단계를 절대 생략하지 않습니다.
(방어 스크립트 전문은 `00_master_router.md`의 [바이브 코딩 차단 방어벽] 참조)

**Phase 1: 복잡도 분석 통보**
> 🔍 기능 분석 결과: 비즈니스 로직이 포함된 고복잡도 기능(XX점)입니다. YAML 설계가 필요합니다.

**Phase 2: YAML 설계도 출력 및 승인 대기**
> 📐 설계 초안입니다. '진행' 또는 '예'를 입력해 주셔야 코드를 생성합니다.

**Phase 3: 명시적 승인 확인 후 코드 생성**
- 반드시 승인 키워드를 받아야 구체적 코딩에 진입합니다.

## 1.4 AI 자가 검증 체크리스트 (코드 생성 전후)
- **순수성 검사**: RA 내부에 `fetch`, `db`, `localStorage`, `창(window)` 접근이 존재하면 즉시 중단하고 분리.
- **컨텍스트(Context) 맵핑**: 모든 데이터 흐름은 `context.input.xxx`, `context.[step]Result.xxx` 표준 준수.
- **보안 검증**: 입력 객체에 항상 `authContext: { userId, roles... }` 포함 확인. (직접 인가는 RA로 분리)

## 1.5 복잡도 계산 예시 (대표 샘플)

### 결제 및 저장 처리
```text
감지 키워드: "결제"(+10), "검증"(+10), "저장"(+10)
구조적 복잡도: DB접근(+15), 트랜잭션(+15), 보안(+10)
총점: 70점 → 고위험 기능이므로 즉시 YAML 설계 강제 진입 
```

## 1.6 요약
- **무조건 설계부터**: 어떤 복잡한 로직이라도 YAML 승인 전에는 `ts/tsx` 코드를 작성하지 않는 것이 이 아키텍처의 핵심입니다. 세부 행동 강령은 `00_master_router`의 방어 지침에 따릅니다.
