# 프로젝트 특수 지침: 배너(Banner) 동기화 규칙

## 📌 이 지침을 읽어야 하는 상황

AI는 다음 키워드 감지 시 이 파일을 **반드시 먼저** 읽는다:
**"배너", "banner", "광고", "가로 세로", "비율", "사이드 배너", "날개 배너"**

---

## 1. 핵심 원칙 (절대 준수)

> **"본문 메인 배너와 사이드 배너의 크기는 비례하며, 가로와 세로 길이만 서로 뒤바뀐 상태로 항상 동일한 비율(1.5 : 1)과 물리적 면적을 유지한다."**

### 1.1 비율 절대 거울 등식 (Golden Ratio Equality)
배너의 크기/비율 조정을 요청받을 경우, 두 타입의 배너 비율이 **역수** 관계로 일치하는지 반드시 확인해야 합니다.
- **메인(본문) 가로 배너**: `aspect-[3/2]` (가로 1.5 : 세로 1 비율) - `premium-job-card.tsx`, `special-job-card.tsx`, `general-job-item.tsx`
- **사이드(날개) 세로 배너**: `aspect-[2/3]` (가로 1 : 세로 1.5 비율) - `side-banners.tsx`

> **[절대 물리적 등식 (Unbreakable Chain)]** ⭐⭐⭐⭐⭐
> "사이드 배너의 가로(Width) 길이 == 홈 바디 메인 배너의 세로(Height) 길이"
> 두 배너의 크기가 변경되더라도 이 등식은 물리적으로 무조건 1px의 오차 없이 같아야 합니다. 이 연결고리를 깨는 수정은 어떤 경우에도 승인되지 않습니다.

### 1.2 날개 배너의 폭(Width) 조작 계산법
CSS Grid 프레임워크 상, 화면이 줄어들면 메인 컨테이너 폭에 비례해 본문 메인 배너의 크기(가로/세로)가 자동으로 작아집니다.
그러나 **사이드 배너는 `fixed` 속성이므로 메인 배너의 높이(세로 길이)를 수학적으로 추적하여, 그 값을 날개 배너의 가로 길이(`Width`)로 명시적 주입해야** 합니다.

**[절대 등식을 유지하는 3단계 계산법]**
1. 메인 배너 Grid Cell 가로 폭 계산: `(컨테이너 폭 - 좌우 패딩 - (Gap 개수 * Gap 너비)) / 열(Cols) 갯수`
2. 메인 배너 Grid Cell 세로 높이 계산: `Grid Cell 가로 폭 / 1.5`
3. ⭐️ 이 계산된 **메인 배너의 세로 높이 값**을 사이드 배너의 **가로 길이(Width)**(`w-[...px]`) 값으로 그대로 적용합니다.

**현재 기준 맵핑:**
- `xl` (1280px 뷰포트 / Container 920px) -> `xl:w-[132px]`
- `2xl` (1440px 뷰포트 / Container 1100px) -> `2xl:w-[127px]`
- `3xl` (1920px 뷰포트 / Container 1500px) -> `3xl:w-[148px]`
- `4xl` (2560px 뷰포트 / Container 2040px) -> `4xl:w-[208px]`

### 1.3 브레이크포인트별 바디(Container) 영역의 물리적 제한 및 마진 클램핑
날개 배너가 나타나는 `xl (1280px)` 이상의 해상도에서는, **날개 배너가 위치할 공간을 확보해야 하므로 메인 컨테이너(Body) 영역의 최대 폭이 뷰포트에 비해 강제로 좁게 설정**되어 있습니다. 이 치수들은 `tailwind.config.ts`의 `theme.container.screens`에 하드코딩 되어 있으며 임의 수정이 엄격히 금지됩니다.

> **[뷰포트 대비 메인 바디 컨테이너 고정 스펙 (tailwind.config.ts)]**
> - **xl** (뷰포트 1280px) → 컨테이너 폭 **최대 920px**
> - **2xl** (뷰포트 1440px) → 컨테이너 폭 **최대 1100px**
> - **3xl** (뷰포트 1920px) → 컨테이너 폭 **최대 1500px**
> - **4xl** (뷰포트 2560px) → 컨테이너 폭 **최대 2040px**

**마진 클램핑 안전 장치**: 이 컨테이너 폭을 임의로 늘릴 경우 날개 배너가 본문 컨테이너 영역과 겹치는 끔찍한 UI 버그(Overlap)가 필연적으로 발생합니다. 위 정해진 컨테이너 크기는 날개 배너와 홈 바디 배너가 충돌하지 않고 중앙 기점 Offset 대비 양쪽에 최소 `15px`의 안전 여백을 확보하기 위해 수학적으로 역산된 절대 수치입니다.

## 2. AI 필수 질문 프로토콜

배너 관련 레이아웃 수정 요청 시 반드시 아래 질문을 먼저 수행한다:

~~~text
⚠️ [프로젝트 특수 지침 적용: banner_responsive_sync_rules]

배너의 크기나 비율 변경을 요청하셨습니다.
본 프로젝트의 규칙에 따라 "메인 배너와 사이드 배너의 크기는 서로 비례(가로/세로 역수)"해야 합니다.

어느 범위까지 변경을 원하십니까?:
1) [내용/스타일만 수정] (비율/크기에는 손대지 않고 색상, 폰트만 변경)
2) [배너 비율 동기화 수정] (배너의 기준 비율을 변경하되 메인/사이드 배너 수식을 모두 재계산하여 동기화)
3) [부분 예외 수정] (예외적으로 모바일에서만 크기를 변경하는 경우)

선택: (1/2/3)
~~~

**옵션별 처리 방식:**
- **옵션 1 (내용/스타일만 수정)**: 기존 `aspect-` 비율 클래스를 절대 유지한 채 내부 내용만 수정합니다.
- **옵션 2 (배너 비율 동기화 수정)**: 새로운 비율(예: 2:1)이 요구되면 `00_master_router.md`의 구조계산법에 따라 컨테이너 Max-Width와 오프셋, Wing Width를 전면 재계산하는 **YAML 설계서**를 제시해야 합니다.
- **옵션 3 (부분 예외 수정)**: 해당 특정 해상도에서만 `[vw]` 또는 % 단위로 처리하되, 사이드 배너가 숨겨지는 구간인지 먼저 확인합니다.

## 3. 금지 사항 (Never Do)

~~~text
❌ 절대 금지 목록:
- 메인 배너의 비율(`aspect-[]`)을 수정하면서 사이드 배너를 수정하지 않는 행위
- 사이즈 계산 없이 사이드 배너에 임의의 고정 Width(예: `w-[150px]`)를 넣는 행위
- 사이드 배너의 간섭 여부 계산 없이 `tailwind.config.ts`의 `container` 폭을 넓히는 행위
~~~
